# 蜂巢 Hive · 答辩指南

> 一句话介绍：**蜂巢是一个 Discord 风格的实时聊天社区**——Spring Boot + WebSocket + MyBatis + MySQL，支持多社区、无限嵌套的"群中群"频道、私聊好友、位掩码角色权限、15 个成就与全屏彩蛋，单 jar 部署、自带便携数据库、开箱即用。

## 一、5 分钟演示脚本

1. **启动**（10 秒）：双击 `start-hive.bat` → 自动拉起 MySQL + 应用。强调：*首次启动自动建库建表（`createDatabaseIfNotExist` + 幂等 schema.sql），评委机器上不用装任何环境*。
2. **登录页**（20 秒）：展示深夜蜂房设计：六边形 LOGO、漂浮粒子动画。登录 `afeng / 123456`。
3. **群中群**（40 秒）：左侧频道树 → 「学习专区」里套「卷王小分队」再套「深夜自习室」——*数据库 channels 表 parent_id 自引用实现树形，服务端限深 5 层防套娃*。
4. **实时消息**（60 秒）：开第二个浏览器窗口登录 `xiaomi`，两窗对发消息：实时到达、正在输入提示、回复引用、表情回应、撤回、图片、未读红点。
5. **彩蛋炸场**（40 秒）：发 `/roll`、`/fortune`；再发"祝大家答辩顺利 🎉"→ **全屏彩带**；按 ↑↑↓↓←→←→BA → **蜜蜂雨 + 隐藏成就金色弹窗**。
6. **权限管理**（40 秒）：用 afeng 给 xiaomi 禁言 10 分钟 → xiaomi 发言被服务端拦截（红色提示）；解除后创建"巡逻蜂"角色（勾选管理频道）分配给 xiaomi → xiaomi 立刻能建频道，名字变色。
7. **私聊好友**（30 秒）：🍯 主页 → 好友申请/接受实时弹通知 → 私聊。
8. **数据可视化**（30 秒）：🏆 成就墙、个人资料里的聊天热力图、📊 蜂巢活跃统计、🔍 中文全文搜索"答辩"。

## 二、技术亮点讲解卡（评委问到就展开）

| 亮点 | 在哪 | 怎么讲 |
|---|---|---|
| **手写 JWT（HS256）** | `util/JwtUtil.java` | 不依赖第三方库：`base64url(header).base64url(payload).HMAC-SHA256`；验签用 `MessageDigest.isEqual` 常量时间比较防时序攻击；单测覆盖篡改/拼接伪造/过期 |
| **BCrypt 密码** | `AuthService` | 数据库只存加盐哈希，登录统一报"用户名或密码错误"不暴露用户存在性 |
| **位掩码权限** | `common/Permissions.java` | Discord 同款：每个权限占一个二进制位，角色权限按位 OR 合并，一条 `BIT_OR` 聚合 SQL 算出生效权限（`RoleMapper.effectivePermissions`） |
| **群中群** | `channels.parent_id` 自引用 | 树形外键 + 服务层深度限制；删分区时子频道上移一层而非级联误删 |
| **WS 广播挂事务 afterCommit** | `ws/WsPush.java` | 实测踩坑：广播先于事务提交，接收方回查读不到数据（回复引用丢失）→ `TransactionSynchronization.afterCommit` 根治，回滚也不会发幽灵消息 |
| **邀请码原子核销** | `InviteMapper.consume` | 次数/有效期校验放进 `UPDATE ... WHERE` 条件，并发抢最后一个名额不超发（影响行数判定） |
| **观察者模式成就引擎** | `event/` + `AchievementService` | 业务只发 Spring 事件，成就规则全部在监听器里，零侵入可扩展；`INSERT IGNORE` 幂等防重复解锁 |
| **中文全文搜索** | `messages` 表 `FULLTEXT ... WITH PARSER ngram` | MySQL 原生 ngram 分词器，`MATCH AGAINST` 自然语言模式 |
| **utf8mb4** | 建表 DDL | 4 字节编码完整支持 emoji 表情存储 |
| **软删除** | `messages.deleted` | 撤回不物理删除，保留数据可审计 |
| **未读数** | `read_states` + 聚合 SQL | 每人每频道记 `last_read_message_id`，一条 GROUP BY 算出全部频道未读 |
| **并发安全推送** | `ChatWebSocketHandler` | `ConcurrentWebSocketSessionDecorator` 包装会话 + `ConcurrentHashMap` 注册表，支持多端登录 |

## 三、高频问题与答法

**Q：为什么不用 Spring Security？**
A：本项目鉴权模型简单（登录态 + 业务内权限位），只引入它的 crypto 模块做 BCrypt；JWT 校验用一个 `HandlerInterceptor` 完成，链路全部自己可控、可讲解。完整引入 Spring Security 反而掩盖了原理。

**Q：为什么用 MyBatis 不用 JPA？**
A：课程重点是数据库应用能力。MyBatis 手写 SQL 让树形查询、BIT_OR 聚合、全文检索、原子核销这些 SQL 设计点全部显式可见；JPA 自动生成 SQL 不利于展示对数据库的掌握。

**Q：消息的实时性怎么实现的？**
A：原生 WebSocket + 自定义 JSON 信封协议（`{type,data}`）。握手阶段用 URL 上的 JWT 认证；服务端维护 用户→会话集合 注册表；蜂巢消息推全巢在线成员，私聊只推两位参与者。心跳 25 秒、断线指数退避重连。

**Q：怎么防 SQL 注入 / XSS？**
A：SQL 全部走 MyBatis `#{}` 预编译参数；前端渲染消息先 HTML 转义再做链接/@提及替换，innerHTML 只插入转义后的内容。

**Q：并发下两个人同时用最后一个名额的邀请码会怎样？**
A：核销是一条带条件的 UPDATE，数据库行锁保证只有一个事务影响行数为 1，另一个直接失败返回"已达使用上限"。

**Q：禁言是怎么生效的？**
A：`hive_members.muted_until` 时间戳，发消息时服务端校验 `muted_until > now` 即拒绝——拦截在服务端而非前端，绕过 UI 也没用。
